# 講義スライドの reveal.js 化 (Pandoc 経由)

## メタ情報

- **状態**: abandoned (不採用)
- **作成日**: 2026-05-29
- **最終更新**: 2026-05-29

## 決定 (2026-05-29): 不採用

負担対効果を再評価した結果, **不採用**とした.

- ユーザの真の目的は「授業で投影 + 学生に PDF 配布」であり, この 3 要件
  (スライド単位スクロール / クリッカー送り / PDF) は **PPT が既にネイティブに
  満たしている**. 移行しても新しい価値はほぼ生まれない.
- 一方コストは, 後述「サンプル資料分析」のとおり数式の手 LaTeX 化と図・吹き出し・
  SmartArt の再構築/画像化に支配され, **移行は負担の純増**になる.
- 図表オーサリングは PPT/Keynote の GUI の方が CSS/SVG/Mermaid 記述より速い.

**今後 reveal.js 化が黒字になりうる条件** (再検討トリガー):
図・数式が少ないテキスト主体の講義を, Web で検索可能・リンク可能なコンテンツとして
ブログに統合したくなり, かつ繰り返し編集・保守する場合. 今回の確率スライドは逆側.

以下の分析・設計は, 上記条件で再検討する際の参照用に残す.

## 概要

現在 PowerPoint で作成している授業スライドを Markdown 単一ソースに移行し,
既存の Hakyll + Pandoc パイプラインに `reveal.js` 出力ルートを追加して
「スライド単位スクロール + プレゼンリモコン (レーザーポインター) 送り」に対応した
HTML スライドとして配信する. 同一ソースから配布用 PDF も生成・ダウンロード提供する.

## 動機

- PPT はバイナリで Git 管理に乗らず, 差分・レビュー・複数セッション編集に不向き.
- 講義資料 (`lectures/fp/*.md`, `lectures/slds/*.md`) は既に Markdown + Pandoc で
  管理されており, スライドも MD 化すれば**単一ソース・同一リポ**で扱える.
- 要件である「スライド単位スクロール」「Mac 接続のプレゼンリモコンで送る」は
  HTML スライドフレームワーク (reveal.js) の標準機能であり, 新規開発がほぼ不要.
  - 市販クリッカー (Logicool R400/R500/Spotlight, ELECOM 等) は macOS からは
    Bluetooth/USB キーボードに見え, `PageDown`/`→` (送り), `PageUp`/`←` (戻り) を送る.
  - reveal.js はこれらをデフォルトでバインド済み → 設定ゼロで送り動作.
  - Spotlight 系の画面上ハイライトは Logi Options+ 側機能で全画面の上で効く
    (ブラウザ表示の reveal.js でも問題なし).
- PPT 超えの利点: 発表者ビュー (`S` キーでノート + 次スライド + タイマー),
  fragment (クリックで箇条書き順次表示), PDF 書き出し.

## 設計

### 出力経路 (A案 = Pandoc → reveal.js)

- Pandoc の `-t revealjs` 出力を利用する. 講義 MD と同じ Pandoc 拡張・整形
  (コードブロック, `::: note`, 数式) を流用できる.
- Hakyll に **reveal.js 専用のルール + テンプレート** を 1 本追加し,
  ブログの chrome を被せない全画面 HTML を別ルートで出力する.
  (通常の講義ページとは別の出力先・別テンプレート)
- スライド区切りは見出しレベル or `---` 区切りで制御.

### PDF ダウンロード

同一 MD から配布用 PDF を生成し, 講義ページにダウンロードリンクを置く.
経路は以下から選択 (PoC で確定):

| 経路 | 仕組み | 長所 | 短所 |
| --- | --- | --- | --- |
| ① ブラウザ印刷 | `?print-pdf` + ブラウザ PDF 保存 | 手軽 | 手動 |
| ② decktape | ヘッドレス Chromium で HTML→PDF | 投影と**同一デザイン**, 自動化可 | Chromium 依存 |
| ③ Pandoc beamer | 同 MD を LaTeX 経由で PDF | 既存 xelatex 経路を流用, 依存きれい | 見た目が reveal と別系統 |

- 見た目を投影と揃えたい → ②decktape.
- 依存を増やしたくない・既存 xelatex を活かす → ③beamer.
- 本リポは既に Hakyll から `xelatex` を呼んでいる (`src/Main.hs` の `xelatex`) ため,
  ビルド中の外部 CLI 呼び出しには前例がある.
- fragment を含む場合, 配布 PDF では分解しない方が見やすい
  (`pdfSeparateFragments` を false 相当に).

## サンプル資料分析 (統計学入門 第6回「確率の基礎」, 全 50p)

`tmp/` 経由で受領した実資料 (PPT 由来 PDF, 50p) の前半を精査した結果.
PPT スライド固有の HTML 化障害を実体ベースで棚卸しした.

### 大前提: 自動変換は効かない

- **Pandoc に pptx リーダーは無い** (docx は読めるが pptx は書き出し専用).
  「pptx → md 自動変換」の綺麗な経路は存在しない.
- 現実的な作業は **MD で手再構築 + 素材 (画像/数式) の個別抽出**.
- 素材抽出・数式取り出しは PDF より **.pptx 原本から行う方が綺麗** (写真・グラフの
  抽出, 数式 OMML の取り出し).

### 障害要素の棚卸し (難度別)

| 要素 | 実体 | HTML 化 | 難度 |
| --- | --- | --- | --- |
| 本文・箇条書き・色文字強調 (赤字「>>>」教員コメント) | テキスト | MD. 赤字は CSS class (意味的 callout に昇格可) | 容易 |
| 丸囲み数字 ①〜⑧, 上付き | Unicode | フォント依存だが表示可 | 容易 |
| 写真 (サイコロ背景, ベイズ/ラプラス肖像) | 埋め込みラスタ (抽出可) | `data-background-image` / `<img>` | 容易 |
| 色付き見出しバー / サイドバー型レイアウト | PPT 図形/SmartArt | CSS (flex/grid + 背景色) で再現 | 容易〜中 |
| データ表 (数式入り多段ヘッダ) | 表 | HTML table + セル内 MathJax, `colspan` | 中 |
| 数式 (インライン+ディスプレイ; 相関式/偏相関式/公理/Σ/集合記号) | CambriaMath 数式オブジェクト | MathJax/KaTeX 用に **LaTeX へ手再エンコード** (総量が多い) | 中〜重 |
| データ図 (散布図×3, 大数の法則の折れ線) | 埋め込みラスタ | 抽出して貼る or matplotlib/SVG 再生成 | 中 |
| アイコン (コイン, サイコロ線画) | ベクタ/小ラスタ | 抽出 or Iconify 差し替え | 容易〜中 |
| 概念図・ツリー (マインドマップ, プロセス tab の SmartArt) | ベクタ図形 (抽出不可・テキスト化不可) | **Mermaid で再構築可** (ツリーは `graph LR` に直写) or SVG 再描画 | 重 (一部 Mermaid で軽量化) |
| 吹き出し (尻尾付き speech bubble, 波括弧注釈) | PPT オートシェイプ | MD 非対応. CSS div + 三角 tail / SVG / 図ごと画像化 | 重 |

### 3 大障害への対処: ページ単位トリアージ

ページを 3 種に分類して処理する.

1. **テキスト型 (大半)** → MD/reveal で素直に. MD 化の最大の旨味 (差分・検索・PDF).
2. **数式型** → MathJax. ただし **LaTeX への手起こし総量が「地味に重い」最大コスト**.
   自動変換に期待しない.
3. **図型** → 以下 3 択を**ページ単位**で選ぶ:
   - (A) 完全再構築: 吹き出し→CSS div, ツリー→Mermaid, プロット→matplotlib/SVG.
     保守性◎ 工数大.
   - (B) 図だけ画像化: 図領域を PNG/SVG 書き出し, 本文はテキスト. **基本線 (推奨)**.
   - (C) スライド丸ごと画像化: 移植コスト最小だが検索/コピー/PDF テキスト/
     アクセシビリティを全て失う. **最後の手段**.

   例: マインドマップは Mermaid 化で「重い SmartArt」が保守可能なテキストになる.
   逆に数式+吹き出し+波括弧が密集したページは (A) が高コストで (B) が無難.

### コスト集約点

- 真のコストは ① **数式の手 LaTeX 化の総量**, ② **吹き出し/SmartArt ページの
  再構築 or 画像化判断** の 2 点.
- 本資料の比率はおおよそ「テキスト 7 : 数式 2 : 図 1」. テキスト主体ページは
  MD 化が非常に有効.

## ロードマップ

1. **PoC (1 章)**: 講義 1 本を reveal.js 用 MD として用意 (既存 lecture MD 1 本の
   スライド化でも可). サンプル資料を使う場合は **テキスト型 1 + 数式型 1 (p9/p16)
   + 図型 1 (p11 ツリー or p8 散布図)** を 1 本にまとめると全パターンの工数を測れる.
2. Hakyll に reveal.js 出力ルート + テンプレートを追加.
3. **実機確認**: 手持ちのプレゼンリモコンで 送り/戻り/発表者ビュー を検証.
4. PDF 生成経路 (②/③) を 1 つ選び, ダウンロードリンク設置まで自動化.
5. 問題なければ横展開. PPT 由来スライドを順次移植.

## コスト見積もり

- PoC (ルート追加 + テンプレート + 1 章スライド化): 小〜中.
- PDF 自動化: ② は Chromium 導入込みで中, ③ は既存経路流用で小〜中.
- PPT からの移植: **本数に比例 (lossy・手作業)**. 下記リスク参照.

## 未解決事項 / リスク

- **PPT → MD は不可逆・lossy**: 複雑レイアウト, 細かい位置調整, 凝ったアニメ,
  埋め込みオブジェクトは移植ではなく再構築になる. 図表・コード・数式中心の
  講義なら相性良い.
- **画像管理**: PPT 内の図は手動で書き出し `images/` 管理が必要.
- **アニメーション**: build-by-click は reveal の fragment で再表現 (自動変換なし).
- **「1 枚に詰める」スライドは MD と相性が悪い**. MD 化を機に情報設計を整理する前提.
- **PDF と投影でデザインを揃えるか**: ②(揃う) か ③(別系統) かの方針決定が必要.
- プレゼンリモコンの具体的な送出キーは型番依存 → 実機確認で確定.
