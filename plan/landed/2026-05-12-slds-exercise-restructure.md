# slds 講義への演習見出し規約 + 回答埋め込みの適用

## メタ情報

- **状態**: landed (Phase 0, 1, 3, 5 完了 / Phase 2, 4 はスコープ外として撤回)
- **作成日**: 2026-05-12
- **最終更新**: 2026-05-12 (実施完了反映)

## 概要

fp 講義 ([plans/in-progress/fp-examples-verification.md](fp-examples-verification.md) で確立した規約) と同様に, slds 講義の各演習にも:

1. **`### Exercise SLDSx-y` 形式の見出し** (ToC 露出 + ASCII アンカー)
2. **章ファイル内に `<details class="protected">` で回答を埋め込む**

を適用する.

なお, 現行の **[slds-answer.md](lectures/slds/slds-answer.md) は学生が作成したもので品質が低い** とオーナー判断があるため, **コンテンツ移行はせず削除方針** とする. 各演習の回答は **教員が新規に書き起こす** 方針.

## 動機

### fp 側で得られた成果

- 演習が ToC で一覧化されて学習者がナビゲートしやすい
- アンカーが `#exercise-chN-K` で URL 共有が簡単
- 回答が同一ファイルに `<details>` で隠れているため, 章を読みながらすぐに自己採点できる
- 提出ファイル名 `chN-K.hs` とアンカーが 1:1 対応

### slds 側の現状の課題

- **演習見出しが不統一**: 確認できた `**演習**` ブロックは **3 章 (slds3, slds4, slds6) のみ計 10 件**. 他章には演習マーカーが無い (実際には演習として運用しているはずだがフォーマットで明示されていない).
- **既存の回答ファイル ([slds-answer.md](lectures/slds/slds-answer.md), 1457 行) は学生作成で品質が低い**:
    - **コンテンツの信頼性が低い** ためそのまま教材化できない
    - 番号体系 (`# 3.3 演習`, `# 6.4.2.4 演習` 等) が **テキスト節番号** ベースで, slds<N>.md の章番号と一致しない
    - **同番号の重複が 23 件** ある (年度違いと推定)
    - 章ファイルから回答ファイルへのリンクは [slds1.md:88](lectures/slds/slds1.md#L88) の 1 箇所のみ
- **既存回答ファイルは削除方針**: コンテンツ移行はせず, 教員が必要な演習について新規に回答を書き起こす.

### 回答ファイル側のセクション

[slds-answer.md](lectures/slds/slds-answer.md) は H1 (`#`) 見出しで 54 セクション. うち **重複が 23 件** (同じセクションが 2 回出現). ユニークなセクション番号は約 31 件.

代表例:
- `# 3.3 演習`, `# 3.4 演習`, ..., `# 3.9 演習` (基礎 Python)
- `# 5.9 演習`, `# 5.10 演習`, ..., `# 5.17 演習①`, `# 5.17 演習②`
- `# 6.3 演習`, `# 6.4.1 演習`, `# 6.4.2 演習`, `# 6.4.2.4 演習`, `# 6.4.2.5 演習`
- `# 9.1-9.2 演習`, `# 9.3-9.4 / 9.10-9.12 演習`, `# 9.5 演習`

→ **番号体系がテキスト節番号 (4 階層まである)** で, slds<N>.md の章番号 N とは完全に独立した別軸. たとえば `3.3` は slds3 ではなくテキスト「3 章 3 節」の意味と推定される.

### 数の不一致

- 章ファイル側の演習マーカー: 10 件
- 回答ファイル側のユニーク回答: 約 31 件
- **差分 21 件** は「章ファイル側で演習として明示されていないが実質演習として運用されている記述」と推定される.
- ただし回答ファイルは削除対象なので, **差分の演習を章ファイル側で改めて掘り起こす** 必要がある (教員主導).

## ゴール / 非ゴール

### ゴール

1. すべての slds 章で演習が **`### Exercise SLDSx-y`** 形式の H3 見出しを持つ (ToC 露出).
2. 各演習の直後または直下に `<details class="protected" data-pass="...">` で **教員が書き起こした回答が同一ページ内に埋め込まれる** (fp と同様の UX). 回答は **slds-answer.md からの転載ではなく新規作成**.
3. **既存の slds-answer.md を削除**. [slds1.md:88](lectures/slds/slds1.md#L88) の外部リンクも削除.
4. CLAUDE.md に slds の演習見出し規約を追記.

### 非ゴール

- **slds-examples** のような検証用 Python プロジェクトの新規作成は対象外 (fp は型システムによる検証価値が高いが, slds は Python + データ分析で検証コストが高いため別計画として扱う).
- 既存講義の **内容そのもの** の修正・追加 (typo 等の発見ベースでの修正は許容).
- テキスト節番号 (`3.3` 等) と章ファイル名 (`slds3` 等) の関係を完全に明らかにすること (これは別途確認).

## 主要な論点

### 論点 1: 番号体系の選択

slds の演習番号は「テキスト節番号」をベースにすると **章をまたいで連続** (3.3, 3.4, ..., 5.9, 5.10, ..., 9.5) するが, 章ファイル名 `slds<N>.md` とは別軸. fp の `CH<n>-<k>` 形式を当てはめるには以下の選択肢がある.

| 案 | 形式 | アンカー | 利点 | 欠点 |
|---|---|---|---|---|
| **A. 章番号ベース連番** | `Exercise SLDS3-1`, `SLDS3-2`, ..., `SLDS4-1`, ... | `#exercise-slds3-1` | fp と同じ規約. 学生に分かりやすい | 既存 slds-answer.md の節番号と乖離 |
| **B. テキスト節番号を維持** | `Exercise 3-3`, `Exercise 5-17a`, `Exercise 6-4-2-4` | `#exercise-3-3`, `#exercise-6-4-2-4` | 既存番号を保持. テキストと参照しやすい | 学生がどの章ファイルかすぐ分からない. アンカーが深い階層 |
| **C. ハイブリッド** | `Exercise SLDS6-1` の見出しに「**(テキスト 3.6 節)**」を併記 | `#exercise-slds6-1` | 両者の参照可能 | 表記が二重で煩雑 |

**推奨: 案 A**. 理由:
- fp と統一規約になることで「Exercise XXX-N」というブランドが講義シリーズ全体で確立する
- アンカー URL が短く ASCII で安定
- 学生が「`slds4.md` の Exercise SLDS4-3」を `slds4-3.py` として提出する直接対応が成立
- 既存テキスト節番号は **見出し本文 / 本文中の参照テキストに残す** ことで対応可

### 論点 2: 回答埋め込みの方針

**前提**: 既存 [slds-answer.md](lectures/slds/slds-answer.md) は学生作成で品質が低いため **コンテンツ移行はしない**. 教員が新規に書き起こす.

| 案 | 内容 | 利点 | 欠点 |
|---|---|---|---|
| **A. 章内に `<details class="protected">` で埋め込み (新規作成)** | fp と同じ構造. 回答は教員が新規執筆 | 学習導線が章内で完結. 品質統制可能 | 教員工数が大きい (各演習につき回答作成が必要) |
| **B. 演習見出しのみ整備し, 回答は当面 "TODO" としておく** | スキャフォールドのみ用意 | 最小工数で見出し統一可能 | 回答が無いと演習として未完 |
| **C. 段階的アプローチ** | フェーズ毎に回答完成度を上げる | 工数を分散できる | 完成までの期間が長い |

**推奨: 案 C (段階的)**. 具体的には:
- **フェーズ 1**: 既に明示マーカーがある 10 件について見出しを `### Exercise SLDSx-y` に統一. 回答プレースホルダ `**(回答: 後日記載)**` を `<details class="protected">` 内に置く.
- **フェーズ 2**: 演習掘り起こし中の各演習についても同様にスキャフォールドを置く.
- **フェーズ 3 以降**: 教員が `<details>` 内の回答を順次執筆.

### 論点 3: 章ファイル側の演習掘り起こし

`**演習**` マーカーは 10 件のみだが, 回答ファイルには ~31 件. **slds-answer.md にある "本来の演習" を, 章ファイル側に逆輸入する必要** がある.

これは単なるフォーマット適用ではなく, **教材の演習構造そのものを再構築する** 作業になる. 著者 (講義担当者) の判断が必須.

### 論点 4: slds-answer.md の扱い

**方針確定: 削除**. 学生作成のため品質が担保できず, コンテンツを章ファイルに移行する価値も低い.

削除タイミング:
- **保守的**: フェーズ 3 で章ファイル側に十分な回答が揃ってから削除
- **積極的**: フェーズ 1 着手と同時に削除し, [slds1.md:88](lectures/slds/slds1.md#L88) の外部リンクも除去

**推奨: 保守的 (フェーズ 3 で削除)**. 章側の回答スキャフォールドが揃うまでは, 万一の参考情報源として保持する.

なお, ファイル削除時の影響:
- `slds-answer.html` への外部リンクは [slds1.md:88](lectures/slds/slds1.md#L88) からの 1 箇所のみ → 同時更新で OK
- Hakyll の build 設定で `slds-answer.md` が個別にビルドされている場合は site.hs から該当パターンも削除する必要あり (要確認)

## 推奨実装プラン (フェーズ分け)

### フェーズ 0: 規約整備 (半日)

- 本計画文書のレビューと方針確定 (論点 1〜4 の選択を決定)
- CLAUDE.md に「slds 講義の演習見出し規約」セクションを追記 (fp と並列)
- 移行作業のためのテンプレ Snippet を準備

### フェーズ 1: マーカー存在章の見出し統一 (半日 〜 1 日)

最小スコープ. 既に `**演習**` が明示されている **slds3 (2 件), slds4 (7 件), slds6 (1 件)** の 10 件を:

- `### Exercise SLDSx-y` 形式に変換
- 直下に `**<演習タイトル>**` を残す (fp と同形式)
- `<details class="protected" data-pass="...">` ブロックを設置. 中身は **`**(回答: 後日記載)**`** プレースホルダ.
- `slds-answer.md` の対応セクション (学生作成) は **参照しない**

このフェーズで **UI / UX / アンカー / TOC 表示 / `::: note` 内 H3 の挙動** をエンドツーエンドで検証 (fp と同じ Pandoc 挙動が再現されるか確認).

### フェーズ 2: マーカー無し章の演習掘り起こし (3-5 日, 教員主導)

章ファイル側でマーカーが無いが実質演習として運用されている記述を **教員が判断して** 章ファイルに演習ブロックとして整形する.

- 各章を通読し, 演習として運用すべき箇所を特定
- 該当箇所に `### Exercise SLDSx-y` を新規挿入
- `<details>` 内の回答プレースホルダも同時に設置
- **このフェーズはテキスト内容理解と教育判断が必要なため AI 単独不可. 教員主導で AI が補助.**

参考情報: slds-answer.md の旧セクション一覧 (3.3, 3.4, ..., 6.4.2.4, 9.1-9.2, ...) を **章/節の見当を付けるためのインデックス** として活用可 (内容は参照しない).

### フェーズ 3: slds-answer.md の削除と参照リンク除去 (半日)

フェーズ 1+2 で章ファイル側に十分なスキャフォールドが揃った時点で:

- `lectures/slds/slds-answer.md` を **削除**
- [slds1.md:88](lectures/slds/slds1.md#L88) の「演習回答及びデータ」セクションから外部リンクを除去 (または別文言に書き換え)
- Hakyll の site.hs を確認し, `slds-answer.md` 専用のビルドルールがあれば削除
- `docs/lectures/slds-answer.html` を `publish.sh` で再生成時に削除されることを確認

### フェーズ 4: 回答コンテンツの順次執筆 (継続)

各演習の `<details>` プレースホルダを **教員が順次本物の回答に置き換える**. 進捗管理として:

- 演習ごとに「未着手 / 執筆中 / 完成」ラベル付与 (HTML コメントで `<!-- status: todo -->` 等)
- 完成した演習から優先的にコミット

このフェーズは演習数 (推定 30-40) × 1 件あたり 30 分 〜 1 時間 と想定. 全体で **15-40 時間程度** の執筆作業.

### フェーズ 5: ナビゲーション拡充 (任意)

- 各章末尾に「次の Exercise リスト」を自動生成 (Hakyll テンプレ拡張)
- 章をまたいだ演習一覧ページの作成 (任意)

## コスト見積もり

| フェーズ | 工数 | オーナー作業の比率 | 内容 |
|---|---|---|---|
| 0 | 半日 | 高 (方針決定) | 規約整備 |
| 1 | 半日 〜 1 日 | 低 (機械的変換) | マーカー存在章を見出し統一 + プレースホルダ |
| 2 | 3-5 日 | **高** (内容判定要) | 演習掘り起こし |
| 3 | 半日 | 低 (削除作業) | slds-answer.md 削除 |
| 4 | 15-40 時間 | **最高** (執筆) | 回答コンテンツ執筆 |

**最短ルート (見出し統一だけ)**: フェーズ 0 + 1 で **1.5 日**. これだけでも fp と統一規約のスタンプを 3 章に押せる.

**最短ルート (回答プレースホルダ込みで全章整備)**: フェーズ 0+1+2+3 で **5-7 日**. 削除作業まで完了し, 「演習見出し + 空回答ブロック」が全章揃った状態.

**完全完成**: フェーズ 4 完了で **20-50 時間 (累積)**. 回答が全埋め.

## 未解決事項 / リスク

1. **論点 1, 2, 3 の方針決定** がフェーズ 0 で必須. 論点 4 (slds-answer.md の扱い) は **削除で確定済み**.
2. **章ファイル側の演習掘り起こし** (フェーズ 2) はテキスト内容理解が必要なため, AI 一人で進められない. 教員協働必須.
3. **テキストとの整合性**: slds の節番号 (3.3 等) はおそらく統計テキスト (基礎統計学? 統計検定?) の節番号を反映している. その情報が `Exercise SLDS<N>-<K>` の形式変更によって失われないよう, **本文中に「テキスト X.Y 節相当」と注記** することを検討.
4. **slds の演習は Python コード提出 + 解説提出のハイブリッド**. fp の「.hs を提出」とは少し性質が異なる. 提出物の形式統一規約を別途検討する余地あり.
5. **回答の埋め込みパスワード**: fp と同じ `data-pass="yakagika"` を使うか, slds 専用パスワードを設けるかオーナー判断.
6. **slds-answer.md の削除タイミング**: 「保守的 (フェーズ 3)」を推奨したが, 学生作成回答を一切利用しない場合は **積極削除 (フェーズ 1 と同時)** でも問題ない. その場合は web 上のキャッシュなどに残る可能性も含めて判断.
7. **フェーズ 4 (回答執筆) は別人 (教員) の作業**. 計画文書としては「演習見出しスキャフォールド + 削除完了」までを landed 判定とし, フェーズ 4 はトラッキングのための別計画 (`slds-answers-authoring.md`) として切り出す案も検討.
