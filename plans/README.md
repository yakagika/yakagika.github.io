# plans/

このディレクトリは、当リポジトリで進行中・完了済みの **横断的な計画文書** を集約する場所です. 単一の章の編集や typo 修正など短期的な作業ではなく, **複数ファイル・複数セッションをまたぐ計画** をここに置きます.

## ディレクトリ構成

| ディレクトリ | 用途 |
| --- | --- |
| `in-progress/` | 着手中・検討中・提案段階の計画. 1ファイル = 1テーマ. |
| `landed/` | 完了して反映された計画 (アーカイブ). |

## ライフサイクル

1. **作成**: 新しい計画を `plans/in-progress/<kebab-case-name>.md` として書き起こす.
2. **更新**: 状況や設計の変更があれば該当ファイルを上書き. 大きな方針転換は `## 変更履歴` セクションに追記する.
3. **完了**: 実装が終わり、当初の目的が達成されたら `plans/landed/YYYY-MM-DD-<kebab-case-name>.md` に **移動** (日付プレフィックスを付ける). ファイル内の `状態` フィールドも `landed` に更新する.
4. **破棄**: 採用しないと決まった計画は `landed/` に「不採用」状態で移動するか, このディレクトリから外に出して保存する (履歴は git が持つので削除でも可).

## 命名規則

- ファイル名: kebab-case の英数字. 例 `fp-examples-verification.md`.
- `landed/` 内では先頭に完了日 `YYYY-MM-DD-` を付ける. 例 `2026-06-01-fp-examples-verification.md`.

## 文書テンプレート

各計画文書には少なくとも以下のセクションを含める.

```markdown
# <タイトル>

## メタ情報

- **状態**: proposed / in-progress / landed / abandoned
- **作成日**: YYYY-MM-DD
- **最終更新**: YYYY-MM-DD

## 概要

1-2文で要約.

## 動機

なぜこの計画が必要か.

## 設計

実現方法の骨子.

## ロードマップ

段階的な実行計画.

## コスト見積もり

工数感.

## 未解決事項 / リスク

開かれている論点.

## 関連ファイル

リポジトリ内の関連パス.

## 変更履歴

- YYYY-MM-DD: 作成
- YYYY-MM-DD: 〜を更新
```

## エージェントへの注意

- 横断的な作業 (複数章にまたがる改訂や新規プロジェクトの追加など) を始める前に, このディレクトリを `ls` して既存計画を確認すること.
- 計画と矛盾する変更を加える場合は、計画文書側を先に更新するか、矛盾が出る旨を会話で明示する.
